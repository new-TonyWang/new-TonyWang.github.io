renderdoc Hook原理

针对android 10 剖析renderdoc 工作原理

renderdoc 简介

​ 该项目是一个纯c++的项目，通过cmakelists设置针对不同平台的抓帧方案。

​ 该项目分为三大部分:

  1. renderdoc： 提供了针对android,ios,windows,unix四个平台不同的gpu的api的抽帧方案；
  2. qrenderdoc：使用Qt编写的pc端的界面和控制程序；
  3. renderdoccmd： 提供了qrenderdoc和android或者ios的通讯程序。对于android来说它负责接收qrenderdoc发出的adb 命令和存储用于hook gles api的动态链接库VkLayer_GLES_RenderDoc。

GPU api Hook 框架

对于android来说，renderdoc实现了三种hook方案：

  1. android 10或以上使用GLES layer进行hook
  2. android 10一下使用jdwp将so文件注入到app中，然后使用inline hook与PLThook组合进行hook

renderdoc中实现了一套hook的接口（或者是类），使用者可以继承相关的类来实现对不同GPU相关的api的注册。具体的类如下面代码所示

LibraryHook::LibraryHook()//所有子函数的构造都会调用该函数（例如EGLHook）并将子类加入到Libs，并等待注册
{
  LibList().push_back(this);

void LibraryHooks::RegisterHooks()
{
  BeginHookRegistration();//平台相关

  for(LibraryHook *lib : LibList())//遍历libs并注册。android则会调用GLHook,EGLHook和VulkanHook
    lib->RegisterHooks();

  EndHookRegistration();//平台相关，可以在这个函数执行hook
}

void LibraryHooks::RemoveHookCallbacks()
{
  for(LibraryHook *lib : LibList())
    lib->RemoveHooks();
}

void LibraryHooks::OptionsUpdated()
{
  for(LibraryHook *lib : LibList())
    lib->OptionsUpdated();
}

GPU 函数hook的过程

android平台中hook的流程全部都是在LibraryHooks::EndHookRegistration()中进行，流程如下：

  1. 确保需要hook的动态链接库都加载到了内存；
  2. 使用dlopen加载动态链接库，并使用dlsym获得到相关的GPU函数；
  3. 使用inlinehook 截获GPU 函数；
  4. 注册公共库，例如dlopen和dlsym。这一步的目的是防止app使用这两个函数动态加载和gles相关的动态链接库；
  5. 对inlinehook失败的函数使用PLTHook；
  6. 刷新函数链接，进行PLTHook；
  7. 调用与库绑定的回调函数

inlineHook

renderdoc中利用llvm的codegen实现inline hook。它是通过如下几步实现:

  1. 读取所有注册的函数；
  2. 判断是否是华为手机，若是华为手机则需要注册libhwgl.so然后再进行hook
  3. 修改函数开头的两条汇编，引导caller调用hook函数；
  4. 若inlinehook失败则将其加入fallbackhook，之后使用PLTHook
/**
renderdoc/os/posix/android/android_hook.cpp

*/

#include "interceptor-lib/include/interceptor.h"

void PatchHookedFunctions()
{
  RDCLOG("Applying hooks with interceptor-lib");

// see below - Huawei workaround
#if defined(__LP64__)
  LibraryHooks::RegisterLibraryHook("/system/lib64/libhwgl.so", NULL);
#else
  LibraryHooks::RegisterLibraryHook("/system/lib/libhwgl.so", NULL);
#endif

  rdcarray<rdcstr> libs = GetHookInfo().GetLibHooks();
  rdcarray<FunctionHook> funchooks = GetHookInfo().GetFunctionHooks();

  //加载拦截器
  void *intercept = InitializeInterceptor();

  std::set<rdcstr> fallbacklibs;
  std::set<FunctionHook> fallbackhooks;

  for(const rdcstr &lib : libs)
  {
    void *handle = dlopen(lib.c_str(), RTLD_NOW);

    bool huawei = lib.contains("libhwgl.so");

    if(!handle)
    {
      HOOK_DEBUG_PRINT("Didn't get handle for %s", lib.c_str());
      continue;
    }

    HOOK_DEBUG_PRINT("Hooking %s = %p", lib.c_str(), handle);

    std::set<void *> foundfunctions;
    /**
     * funchooks包含dlopen或者dlsym
     */
    for(const FunctionHook &hook : funchooks)
    {
      void *oldfunc = dlsym(handle, hook.function.c_str());

      // UNTESTED workaround taken directly from GAPID, in installer.cpp. Quoted comment:
      /*
            // Huawei implements all functions in this library with prefix,
            // all GL functions in libGLES*.so are just trampolines to his.
            // However, we do not support trampoline interception for now,
            // so try to intercept the internal implementation instead.
            //华为的opengl接口只是它们自己的跳板，他们重新实现了一套HWOpengl
      */
      if(huawei && oldfunc == NULL)
        oldfunc = dlsym(handle, ("hw_" + hook.function).c_str());

      if(GetHookInfo().IsHooked(oldfunc))
        continue;

      if(!oldfunc)
      {
        HOOK_DEBUG_PRINT("%s didn't have %s", lib.c_str(), hook.function.c_str());
        continue;
      }

      HOOK_DEBUG_PRINT("Hooking %s::%s = %p with %p", lib.c_str(), hook.function.c_str(), oldfunc,
                       hook.hook);

      void *trampoline = NULL;

      bool success = InterceptFunction(intercept, oldfunc, hook.hook, &trampoline, &intercept_error);//开始对函数进行hook，使用llvm来生成汇编代码

      if(!hook.orig)
        RDCWARN("No original pointer for hook of '%s' - trampoline will be lost!",
                hook.function.c_str());

      if(hook.orig && *hook.orig == NULL)
        *hook.orig = trampoline;

      if(success)
      {
        HOOK_DEBUG_PRINT("Hooked successfully, trampoline is %p", trampoline);
      }
      else
      {//inlineHook失败的话就加入fallbackhooks，后续的PLTHook会读取该列表
        RDCERR("Failed to hook %s::%s!", lib.c_str(), hook.function.c_str());
        fallbacklibs.insert(lib);
        fallbackhooks.insert(hook);
      }

      GetHookInfo().SetHooked(oldfunc);
    }
  }

  // we still need to hook android_dlopen_ext with interceptor-lib so that we can intercept the
  // vulkan loader's attempts to load our library and prevent it from loading a second copy (!!)
  // into the process.
  // Unfortunately, interceptor-lib can't hook this function so we need to set up the PLT hooking.
  // This is just a minimal setup to intercept that one function.
  GetHookInfo().ClearHooks();
//注册PLTHook
  for(const rdcstr &l : fallbacklibs)
  {
    RDCLOG("Falling back to PLT hooking for %s", l.c_str());
    GetHookInfo().AddLibHook(l);
  }

  for(const FunctionHook &hook : fallbackhooks)
  {
    RDCLOG("Falling back to PLT hooking for %s", hook.function.c_str());
    GetHookInfo().AddFunctionHook(hook);
  }
}

PLTHook

PLTHook的实现是通过编写dl_iterate_phdr()的回调函数实现的，回调函数大致思路如下：

  1. 判断该动态链接库是否已经被hook，如果没有被hook则继续；
  2. 遍历so文件的phr表，找到dysym,strtab等，并从本地读取相同的so文件中的DYNAMIC段，从而找到rol段的偏移；
  3. 结合符号表，字符串表和rel段的偏移修改内存中rel段中函数的地址，并将其修改为为hook函数的地址，这样在其他动态链接库链接到gles相关的库时就会调用hook函数，从而实现就实现了PLTHook

GLES layer

android 10推出了GLES的分层功能用于调试GPU程序。官方介绍见GLES 分层功能。对于每个图层，LayerLoader 会跟踪 AndroidGLESLayer_Initialize 和 AndroidGLESLayer_GetProcAddress 入口点。因此只要实现这两个函数即可。

gles layer的这两个函数的完整定义如下：

void AndroidGLESLayer_Initialize(FunctionTable layer_id, EGLFuncPointer next_gpa);
void *AndroidGLESLayer_GetProcAddress(const char *funcName, EGLFuncPointer next);

运行调试层之前需要使用如下几个命令来启动调试层

# 开启gles 分层功能
adb shell settings put global enable_gpu_debug_layers 1

# 设置需要调试的app
adb shell settings put global gpu_debug_app <package_name>

#输入gles layer所在的so文件的路径，例如"libGLLayer.so"
adb shell settings put global gpu_debug_layers_gles <layer1:layer2:layerN>

# 指定gles layer所在的包的名称
adb shell settings put global gpu_debug_layer_app <package1:package2:packageN>

在完成上述过程之后，app就能够在启动之前通过包名+so文件路径找到gles layer的路径并完成加载。

接下来，我们看一下上述两个函数应该如何正确的使用：

void AndroidGLESLayer_Initialize(FunctionTable layer_id,EGLFuncPointer next_gpa)

该函数又两个参数：

  • layer_id： 不要以为这只是一个layer 的标识符，它其实是一个函数列表。列表中的每一项都表示该层能够调用到的下一层函数或者原函数的地址。它的作用是让本层的的hook函数数跳转到下一层函数或者原函数；

  • next_gpa： 这是一个函数指针，它的完整定义如下：

    void *next_gpa(FunctionTable layer_id,const char* func_name)

    它有如下两个功能：

    1. 通过layer_id和func_name定位到下一层原函数或者原函数；
    2. 通过eglGetProcAddr函数找到扩展函数；

{
  if(RenderDoc::Inst().IsReplayApp())
  {
    if(!EGL.GetDisplay)
      EGL.PopulateForReplay();

    return EGL.BindAPI(api);//调用下一层的BindAPI
  }

  EnsureRealLibraryLoaded();

  EGLBoolean ret = EGL.BindAPI(api);//调用下一层的BindAPI

  if(ret)
    eglhook.activeAPI = api == EGL_OPENGL_API ? RDCDriver::OpenGL : RDCDriver::OpenGLES;

  return ret;
}

  EGL_NONHOOKED_SYMBOLS(EGL_FETCH)

  // 使用相同方法设置gl相关的函数
  GL.PopulateWithCallback(

然后我们看一下另一个函数:

void *AndroidGLESLayer_GetProcAddress(const char *funcName, EGLFuncPointer next_func);

它是起到一个承上启下的作用，用于更新函数指针信息。

当有多个gles layer的时候每一层都需要知道本层layer的函数应该调用到下一层或者驱动层函数的地址，因此该函数就是用于更新暴露给下一层layer的函数地址。使用该函数时只需要根据函数名称给出判断：

  1. 若该层截获了某个函数，则需要返回hook函数的地址；
  2. 若未截获函数则直接返回上一层的地址即可；

结合renderdoc的代码，我们分析一下：

总结

  1. 对于android来说renderdoc使用了两种hook方案：gles layer和jdwp注入；
  2. renderdoc将hook的注册和hook执行进行分离，好处是在不同地方进行注册，最后统一执行hook，并且可以通过不同方式执行hook；
  3. gles layer 可以以so为最小单位，在调用opengles 的draw call之前进行拦截；
  4. 三种hook的优缺点：
inllineHook PLTHook GLES layer
优点 理论可以修改任何代码，非常灵活 实现简单，只需要修改ral段的数据 android原生支持，兼容性很好
缺点 需要修改汇编代码，难度较大。 只能完成lib级别的hook，不能修改lib中的函数调用 仅支持android 10以上的机型，仅限调试是使用

注意事项

  1. gles layer加载的时间是在首次调用opengl函数的时候，使用的方式是dlopen和函数指针；

  2. renderdoc 在系统版本低于android 10的时候使用的是JDWP注入，在java层会调用 Runtime.load()并最终在c++通过libnativeloader.so的dlopen中调用OpenNativeLibrary;

  3. 经测试，对于低于android 10 的华为很大概率最后会使用PLTHook，而非inlineHook。

  4. Renderdoc 不能识别浮点寄存器
