renderdoc 简介

​ 该项目是一个纯c++的项目，通过cmakelists设置针对不同平台的抓帧方案。

​ 该项目分为三大部分:

  1. renderdoc： 提供了针对android,ios,windows,unix四个平台不同的gpu的api的抽帧方案；
  2. qrenderdoc：使用Qt编写的pc端的界面和控制程序；
  3. renderdoccmd： 提供了qrenderdoc和android或者ios的通讯程序。对于android来说它负责接收qrenderdoc发出的adb 命令和存储用于hook gles api的动态链接库VkLayer_GLES_RenderDoc。

GPU api Hook 框架

对于android来说，renderdoc实现了三种hook方案：

  1. android 10或以上使用GLES layer进行hook
  2. android 10一下使用jdwp将so文件注入到app中，然后使用inline hook与PLThook组合进行hook

GPU 函数hook的过程

android平台中hook的流程全部都是在LibraryHooks::EndHookRegistration()中进行，流程如下：

  1. 确保需要hook的动态链接库都加载到了内存；
  2. 使用dlopen加载动态链接库，并使用dlsym获得到相关的GPU函数；
  3. 使用inlinehook 截获GPU 函数；
  4. 注册公共库，例如dlopen和dlsym。这一步的目的是防止app使用这两个函数动态加载和gles相关的动态链接库；
  5. 对inlinehook失败的函数使用PLTHook；
  6. 刷新函数链接，进行PLTHook；
  7. 调用与库绑定的回调函数

inlineHook

renderdoc中利用llvm的codegen实现inline hook。它是通过如下几步实现:

  1. 读取所有注册的函数；
  2. 判断是否是华为手机，若是华为手机则需要注册libhwgl.so然后再进行hook
  3. 修改函数开头的两条汇编，引导caller调用hook函数；
  4. 若inlinehook失败则将其加入fallbackhook，之后使用PLTHook

PLTHook

PLTHook的实现是通过编写dl_iterate_phdr()的回调函数实现的，回调函数大致思路如下：

  1. 判断该动态链接库是否已经被hook，如果没有被hook则继续；
  2. 遍历so文件的phr表，找到dysym,strtab等，并从本地读取相同的so文件中的DYNAMIC段，从而找到rol段的偏移；
  3. 结合符号表，字符串表和rel段的偏移修改内存中rel段中函数的地址，并将其修改为为hook函数的地址，这样在其他动态链接库链接到gles相关的库时就会调用hook函数，从而实现就实现了PLTHook

总结

  1. 对于android来说renderdoc使用了两种hook方案：gles layer和jdwp注入；
  2. renderdoc将hook的注册和hook执行进行分离，好处是在不同地方进行注册，最后统一执行hook，并且可以通过不同方式执行hook；
  3. gles layer 可以以so为最小单位，在调用opengles 的draw call之前进行拦截；
  4. 三种hook的优缺点：
inllineHook PLTHook GLES layer
优点 理论可以修改任何代码，非常灵活 实现简单，只需要修改ral段的数据 android原生支持，兼容性很好
缺点 需要修改汇编代码，难度较大。 只能完成lib级别的hook，不能修改lib中的函数调用 仅支持android 10以上的机型，仅限调试是使用

注意事项

  1. gles layer加载的时间是在首次调用opengl函数的时候，使用的方式是dlopen和函数指针；

  2. renderdoc 在系统版本低于android 10的时候使用的是JDWP注入，在java层会调用 Runtime.load()并最终在c++通过libnativeloader.so的dlopen中调用OpenNativeLibrary;

  3. 经测试，对于低于android 10 的华为很大概率最后会使用PLTHook，而非inlineHook。

  4. Renderdoc 不能识别浮点寄存器
