如何执行.o文件

发表于 2023-08-05 分类于 C++ Disqus：

如何执行.o文件

.o文件是又编译器将.cc或者.c等源代码文件编译而来的，它往往对应一个compilation unit。但是.o文件由于缺少外部符号（调用的外部函数或者全局变量）因此.o文件理论上不可被执行。

但是我们可以通过一些技巧，让.o变得可执行。首先，我们先简单介绍一下.o文件。

.o文件的构成

.o文件也是ELF类型的文件，但是与.so文件不同的是，.o文件仅包含一个compilation unit的信息。我们先介绍一些常用的节，以下图为例：

可以看到里面有很多节，但是我们只需要关注.text,.rela.text,.rodata.xxx(这里指的是.rodata相关的所有节),.data,.symtab和.strtab就行了，下表列举了不同节的作用

.text 存储了所有的汇编指令，但是跳转到.o以外的指令(例如call, b这种指令)使用的跳转地址为0
.rela.text 存储了.text节中所有需要重定位的符号
.rodat.xxx 存储了只读的数据，当有可读可修改的数据的时候，还会有.data节
.symtab 符号表，可以从该表中得到该.o文件中的函数名称
.strtab 字符表，存储了符号表中的符号字符串。

由于架构的不同，x86_86、aarch64和arm32的.o格式不一定完全相同，解析.o文件的流程是几乎一致的。下面就根据这些信息来解析.o文件并执行.o文件。 ## 如何执行.o文件

假设我们要执行.o文件中的某个函数，我们需要进行如下操作：

编写工具函数

由于我们需要解析.o文件，因此需要先写一点功能函数来辅助我们解析。

//寻找通过名称寻找对应的section
const Elf64_Shdr *AkgLibraryLoader::LookupSection(const std::string &name) {
  size_t name_len = name.size();
  /* number of entries in the sections table is encoded in the ELF header */
  for (Elf64_Half i = 0; i < obj.hdr->e_shnum; i++) {
    /* sections table entry does not contain the string name of the section
     * instead, the `sh_name` parameter is an offset in the `.shstrtab`
     * section, which points to a string name
     */
    const char *section_name = shstrtab + sections[i].sh_name;
    size_t section_name_len = strlen(section_name);
    if (name_len == section_name_len && !strcmp(name.data(), section_name)) {
      /* we ignore sections with 0 size */
      if (sections[i].sh_size) {
        return sections + i;
      }
    }
  }
  return nullptr;
}
//解析section_name对应的section，返回section的大小和section在.o文件中的偏移
std::vector<uint64_t> AkgLibraryLoader::ParseSection(std::string section_name, bool is_mandatory) {
  /* find the `.rodata` entry in the sections table */
  const Elf64_Shdr *section_hdr = LookupSection(section_name);
  uint64_t section_size = 0;
  uint64_t section_offset = 0;
  if (!section_hdr) {
    if (is_mandatory) {
      MS_LOG(ERROR) << "Failed to find" << section_name;
      return {};
    } else {
      MS_LOG(INFO) << "Failed to find" << section_name;
    }
  } else {
    section_size = section_hdr->sh_size;
    section_offset = section_hdr->sh_offset;
  }
  return {section_size, section_offset};
}

加载.o文件

这一步很简单，可以通过open()函数和mmap函数将.o文件映射进内存。

解析.o文件

与.so文件相同，.o文件的从0开始的低地址处也是一个ELF header，它的结构对应了如下结构体：

typedef struct
{
  unsigned char	e_ident[EI_NIDENT];	/* Magic number and other info */
  Elf64_Half	e_type;			/* Object file type */
  Elf64_Half	e_machine;		/* Architecture */
  Elf64_Word	e_version;		/* Object file version */
  Elf64_Addr	e_entry;		/* Entry point virtual address */
  Elf64_Off	e_phoff;		/* Program header table file offset。program header对应偏移（该段是用于动态链接的，我们不需要）*/
  Elf64_Off	e_shoff;		/* Section header table file offset，section table对应的偏移，我们需要它*/
  Elf64_Word	e_flags;		/* Processor-specific flags */
  Elf64_Half	e_ehsize;		/* ELF header size in bytes */
  Elf64_Half	e_phentsize;		/* Program header table entry size */
  Elf64_Half	e_phnum;		/* Program header table entry count */
  Elf64_Half	e_shentsize;		/* Section header table entry size */
  Elf64_Half	e_shnum;		/* Section header table entry count ，section table的条目数，和e_shoff 结合使用*/
  Elf64_Half	e_shstrndx;		/* Section header string table index ，section table对应的字符串表，它需要和section table结合使用*/
} Elf64_Ehdr;

我们应用e_shoff对应的偏移就可方访问到section table，其结构具体如下：

typedef struct
{
  Elf64_Word	sh_name;		/* Section name (string tbl index)，在e_shstrndx表中的偏移*/
  Elf64_Word	sh_type;		/* Section type */
  Elf64_Xword	sh_flags;		/* Section flags */
  Elf64_Addr	sh_addr;		/* Section virtual addr at execution */
  Elf64_Off	sh_offset;		/* Section file offset */
  Elf64_Xword	sh_size;		/* Section size in bytes */
  Elf64_Word	sh_link;		/* Link to another section */
  Elf64_Word	sh_info;		/* Additional section information */
  Elf64_Xword	sh_addralign;		/* Section alignment */
  Elf64_Xword	sh_entsize;		/* Entry size if section holds table ，每一个entry的大小，sh_size/sh_entsize 即为符号的数量*/
} Elf64_Shdr;

至此我们就完成了解析执行.o文件时所必要的部分。

.o文件section 装入内存

这一步非常类似.so文件中Program Header中的LOAD段的装入。但是与之不同的是，.o文件中没有Program Header，也没有LOAD段，思考了一下，.text,.data,.rodata这些section其实就对应了LOAD段的内容，因此我们需要将这三部分段进行装入。除此之外，我们还需要额外装入jump table和got table，这个jump table就类似SO文件的PLT/GOT表，got table中保存的都是.o文件内部调用时的函数地址。现在我们详细介绍jump table的功能。

下面结构体对应jump table的一个表项，ext_jump存储的是跳转外部函数的指令。结构体中的instr对应二进制编码的汇编指令，并且不同架构的指令完全不相同，必须分开处理。

  struct ext_jump {
#if defined(ENABLE_ARM64)
     // for arm64 instruction.
    uint32_t instr[5];
#endif
    uint8_t *addr;
    /* unconditional x64 JMP instruction */
    /* should always be {0xff, 0x25, 0xf2, 0xff, 0xff, 0xff} */
    /* so it would jump to an address stored at addr above */
    uint8_t instr[6];
  };
  struct ext_jump *jumptable;
};

在进行装入之前，我们先查询有多少个外部符号和got表项，这样才能正确得开辟内存

constexpr const size_t got_table_item_size = 8;
void AkgLibraryLoader::CountDynmaicSymbols() {
  const Elf64_Shdr *rela_text_hdr = LookupSection(".rela.text");
  if (!rela_text_hdr) {
    MS_LOG(INFO) << "cannot find .rela.text, no need to do relocation.";
    return;
  }
  size_t num_relocations = rela_text_hdr->sh_size / rela_text_hdr->sh_entsize;
  const Elf64_Rela *relocations =
    static_cast<const Elf64_Rela *>(static_cast<const void *>(obj_.base + rela_text_hdr->sh_offset));
  for (size_t i = 0; i < num_relocations; i++) {
    auto symbol_idx = ELF64_R_SYM(relocations[i].r_info);
    auto type = ELF64_R_TYPE(relocations[i].r_info);
    if (symbols_[symbol_idx].st_shndx == SHN_UNDEF) {
      /* 外部符号*/
      num_ext_symbols_++;
    } else if (type == R_X86_64_GOTPCREL || type == R_X86_64_GOTPCREL64) {//x86_64got表项，每一条表项长度是8byte
      /* x86_64 signed PC relative offset to GOT */
      got_table_size_ += got_table_item_size;
    } else if (type == R_AARCH64_ADR_GOT_PAGE) {//aarch64 got表项，每一条表项长度是8byte
      /* aarch64 signed PC relative offset to GOT */
      got_table_size_ += got_table_item_size;
    }
  }
}

  text_runtime_base_ =

  if (text_vec[0] != 0 && memcpy_s(text_runtime_base_, text_vec[0], obj_.base + text_vec[1], text_vec[0]) != EOK) {
    MS_LOG(ERROR) << "memcpy .text section failed, section size:" << text_vec[0];
    return false;
  }
  /* .data will come right after .text */
  data_runtime_base_ = text_runtime_base_ + PageAlign(text_vec[0]);
  if (data_vec[0] != 0 && memcpy_s(data_runtime_base_, data_vec[0], obj_.base + data_vec[1], data_vec[0]) != EOK) {
    MS_LOG(ERROR) << "memcpy .data section failed, section size:" << data_vec[0];
    return false;
  }
  /* .rodata.xxx will come right after .data */
  auto rodata_runtime_base = data_runtime_base_ + PageAlign(data_vec[0]);
  uint8_t *current_rodata_runtime_base = rodata_runtime_base;
  for (size_t idx = 0; idx < rodata_section_list.size(); idx++) {
    section_runtime_base_map_[rodata_section_list[idx]] = current_rodata_runtime_base + page_align_list[idx];
    auto section_size = section_info_map_[rodata_section_list[idx]][0];
    auto section_offset = section_info_map_[rodata_section_list[idx]][1];
    if (section_size != 0 &&
        memcpy_s(current_rodata_runtime_base, section_size, obj_.base + section_offset, section_size) != EOK) {
      MS_LOG(ERROR) << "memcpy " << rodata_section_list[idx] << "section failed, section size:" << section_size;
      return false;
    }
  }
  /* jumptable_ will come at last */
  jumptable_ = static_cast<ext_jump *>(static_cast<void *>(current_rodata_runtime_base + page_align_list.back()));
/* got table */
  got_table_ =
    static_cast<uint8_t *>(static_cast<void *>(jumptable_ + PageAlign(sizeof(struct ext_jump) * num_ext_symbols_)));

.o文件符号重定位

上一步结束，我们就完成了装入的过程。现在需要对.o文件中未定义的符号进行重定位。

为了简化操作，我们直接选取.rela.text section进行重定位，因为该section中存放了所有需要冲行为的信息，如果.o文件不需要重定位的话则该section不存在。

.rela.text section的数据结构如下

typedef struct
{
  Elf64_Addr	r_offset;		/* Address，在.o文件的偏移 */
  Elf64_Xword	r_info;			/* Relocation type and symbol index，重定位信息 */
  Elf64_Sxword	r_addend;		/* Addend, 指令偏移（我理解的是这样）*/
} Elf64_Rela;
//可以通过如下宏来获得r_info的信息
#define ELF64_R_SYM(i)			((i) >> 32)
#define ELF64_R_TYPE(i)			((i) & 0xffffffff)
#define ELF64_R_INFO(sym,type)		((((Elf64_Xword) (sym)) << 32) + (type))

符号重定位的过程有点复杂，大致流程如下：

	T1(计算有多个符号需要进行重定位)-->T2

	T3{是否是未定义的外部符号}--是-->T4
	T3{是否是未定义的外部符号}--否-->T6

	T5(在jumptable 中生成调用外部函数的汇编指令)-->T9

在处理外部未符号的时候，我们其实不知道该符号的来源在哪里，手动编写符号重定向的代码工作量又非常大，因此直接执行.o文件就要求用户对这段程序非常熟悉，用户需要知道.o文件所有的符号来源。举一个例子，假设该.o文件仅依赖了当前.so文件的符号，那么我们就可以使用如下代码来访问当前加载.o文件的.so文件的符号地址：

void *AkgLibraryLoader::LookupExtFunction(const std::string &name) {
  void *lib_handle = dlopen(nullptr, RTLD_LAZY | RTLD_DEEPBIND);
  if (lib_handle == nullptr) {
    MS_LOG(ERROR) << "dlopen handler is nullptr.";
    return nullptr;
  }
  return dlsym(lib_handle, name.c_str());
}

aarch64后端外部符号

aarch64由于是精简指令集，每一个指令长度只有32bit，因此我们不可能将一个64位的地址直接写入一调指令中，于是我们必须另辟蹊径，多写几条指令。首先我们查询了aarch64的指令格式，使用4条mov指令将64位的立即数搬运到X17寄存器中，然后使用BR X17指令进行跳转：

/* arm64 branch instrution.
 * For details, please check
 * https://developer.arm.com/documentation/ddi0596/2021-12/Base-Instructions/MOVK--Move-wide-with-keep-?lang=en
 * The generated instructions in jumptable always like:
 * assume 'addr = 0x0000ffffad688138'
 * mov x17,#0x8138
 * movk x17,#0xad68, lsl #16
 * movk x17,#0xffff, lsl #32
 * movk x17,#0x0, lsl #48
 * br x17
 */
int idx = 0;
constexpr const uint32_t addr_shift = 0x10;
constexpr const uint32_t mask = 0xFFFF;
constexpr const uint32_t movk_x17 = 0xF2800011;
constexpr const uint32_t hw_shift_size = 21;
constexpr const uint32_t br_x17 = 0xD61F0220;
constexpr const int shift_size = 4;
constexpr const int addr_mask_shift_size = 5;
for (int shift = 0; shift < shift_size;) {
  auto masked_imm16_shift = (addr & mask) << addr_mask_shift_size;
  if (idx == 0) {
    movk = 0xD2800011;
  }
  uint32_t hw = shift << hw_shift_size;
  jumptable[curr_jmp_idx].inst[idx] = movk_x17 ^ hw ^ masked_imm16_shift;
  idx++;
  shift++;
  addr = addr >> addr_shift;
}
jumptable[curr_jmp_idx].instr[idx] = br_x17;

指令补全或指令重写

上文提到，我们需要计算出该数据距离vbroadcastss 0x0(%rip),%ymm1指令的偏移，因此通过下面的代码计算从指令到数据、jump table以及got table的具体表项的偏移，并将指令补全或重写。

switch (type) {
  case R_AARCH64_ADR_GOT_PAGE: { //aarch64 adrp 指令，该指令用于跨页获取地址，该指令会获取目标页得基址，用于获取got项页的基址
    /* P-page-rel using adrp instruction */
    constexpr const uint64_t page_bits = 0xFFFFFFFFFFFFF000;
    constexpr const uint64_t distance_shift = 12;
    constexpr const uint64_t immhi_shift = 5;
    constexpr const uint64_t immlo_shift = 29;
    auto base = reinterpret_cast<uint64_t>(patch_offset) & page_bits;
    auto got_table_page = reinterpret_cast<uint64_t>(got_table_ + got_table_offset) & (~(page_size_ - 1));
    auto distance = (got_table_page - base) >> distance_shift;
    /* low 2 bits in distance */
    auto immlo = distance & ((1 << 2) - 1);
    /* high 18 bits in distance */
    auto immhi = (distance >> ((1 << 2) - 1)) & ((1 << 17) - 1);
    /* patch the adrp instruction */
    *(static_cast<uint32_t *>(patch_offset)) |=
      (static_cast<uint32_t>(immhi << immhi_shift) | static_cast<uint32_t>(immlo << immlo_shift));
    *(static_cast<uint64_t *>(static_cast<void *>(got_table_ + got_table_offset))) =
      reinterpret_cast<uint64_t>(symbol_address);
    break;
  }
  case R_AARCH64_LD64_GOT_LO12_NC: {//aarch adrp指令的后续指令，使用 ldr 指令将页内偏移送给寄存器，用于获取got的页内偏移
    /* Patch the ldr instruction */
    constexpr const uint32_t page_offset_shift = 10;
    auto page_offset = reinterpret_cast<uint64_t>(got_table_ + got_table_offset) & (page_size_ - 1);
    *(static_cast<uint32_t *>(patch_offset)) |= static_cast<uint32_t>(page_offset << page_offset_shift);
    got_table_offset += got_table_item_size;
    break;
  }
  case R_AARCH64_CALL26: { // aarch 64 的bl指令，用于外部函数调用，该指令会访问jumptable, symbol_address就是从jumptable中获得到的
    /* This can jump to +/-128MB. We are able to this because jumptable is close to .text */
    constexpr const uint32_t aarch64_call_addr_shift = 2;
    *(static_cast<uint32_t *>(patch_offset)) ^=
      (static_cast<uint32_t>((static_cast<uint8_t *>(symbol_address) + relocations[i].r_addend -
                              static_cast<uint8_t *>(patch_offset))) >>
       aarch64_call_addr_shift);
    break;
  }
  case R_X86_64_GOTPCREL:// x86_64 中32位 plt/got指令，用于.o文件内部函数调用，该指令会方位got表
  /* 32 bit signed PC relative offset to GOT , which follows G + GOT + A - P */
  case R_X86_64_GOTPCREL64:// x86_64 中64位 plt/got指令，用于.o文件内部函数调用，该指令会方位got表
    /* 64 bit signed PC relative offset to GOT , which follows G + GOT + A - P */
    *(static_cast<uint32_t *>(patch_offset)) =
      got_table_ + got_table_offset + relocations[i].r_addend - static_cast<uint8_t *>(patch_offset);
    *(static_cast<uint64_t *>(static_cast<void *>(got_table_ + got_table_offset))) =
      reinterpret_cast<uint64_t>(symbol_address);
    got_table_offset += got_table_item_size;
    break;
  case R_X86_64_PC32: // x86_64 获取data段数据地址的指令
    /*  For .rodata.xxx symbols */
  case R_X86_64_PLT32: {// x86_64 call指令，用于外部函数调用，该指令会跳转到jumptable
    /*  Change 'call 0000' into 'call &jumptable' for external symbols */
    *(static_cast<uint32_t *>(patch_offset)) = static_cast<uint32_t>(
      static_cast<uint8_t *>(symbol_address) + relocations[i].r_addend - static_cast<uint8_t *>(patch_offset));
    break;
  }
  default: {
    MS_LOG(ERROR) << "Can not support relocation type:" << type
                  << ", whose symbol is : " << strtab_ + symbols_[symbol_idx].st_name;
    return false;
  }
}

到此为止，整个.o文件重定向的功能就完成了。但是上述重定向做的还是不够充分，暂时未考虑如下两个重定向。在x86_64执行时，不会core，但是执结果会有问题。

执行.o文件

最困难的工作，我们已经完成了，接下来就需要执行.o文件了。这一步相对简单，我们实现一个和ldsym()功能相近的函数即可，这样就可以用该函数获取.o文件的函数地址。

void *AkgLibraryLoader::LookupFunction(const std::string &name) {
  size_t name_len = name.size();
  /* loop through all the symbols in the symbol table */
  for (int i = 0; i < num_symbols; i++) {
    /* consider only function symbols */
    if (ELF64_ST_TYPE(symbols[i].st_info) == STT_FUNC) {
      /* symbol table entry does not contain the string name of the symbol
       * instead, the `st_name` parameter is an offset in the `.strtab`
       * section, which points to a string name
       */
      const char *function_name = strtab + symbols[i].st_name;
      size_t function_name_len = strlen(function_name);
      if (name_len == function_name_len && !strcmp(name.data(), function_name)) {
        /* st_value is an offset in bytes of the function from the
         * beginning of the `.text` section
         */
        return text_runtime_base + symbols[i].st_value;
      }
    }
  }
  return nullptr;
}

拿到函数指针之后，直接执行该函数就可以了。

使用场景

上面说了这么多，大家想知道究竟什么场景才适合用这种方式嘛。首先，这种方式可以避免dlopen，因为有的时候，我们可能会把so或者.o文件存储在二进制文件中，但是想使用dlopen的时候就必须把so文件从二进制数据中dump出来才行，因此我们可以另辟蹊径得执行.o程序，这样可与直接在内存中操作。除此之外，也可以用于没有静态链接器的环境，因为有的时候用户现场只有运行程序的环境，并不需要g++，gcc这种开发工具，因此也可以使用该方式。

真实场景

在能图算融合的Mindspore Lite中就可以使用该方式。图算融合会使用调用akg，然后akg会调用llvm生成算子内核函数对应的.o文件，该.o文件的内容akg完全可控，并且算子内核函数较为简单，我们可以直接将.o文件打包进ms离线模型中，并在

Mindspore Lite的运行时环境中直接执行.o文件中的算子内核函数。
